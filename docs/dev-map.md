# 名人堂百科开发指南

## 零、开发目的与预期目标

实现各届成员间的沟通，促进不同成员之间的交流，维持成员活跃。以中心化的存储与去中心化的更新方式，实现易维护的个人百科。

注册后，成员可以在网站上自行更新自身的履历与奖项，可选地提交证明材料以提高真实性，交由管理员审核后维护自身的个人百科，并在首页快捷地查询其他当届、往届成员的百科，实现跨届的信息交流与进一步的合作。

## 一、结项要求

### 必选

1. 完成可浏览的多页面网站，包括首页、搜索、个人资料页、动态页和设置页等。

2. 实现多类型用户分层权限，允许管理员对个人百科进行管理，用户更新个人百科需要交由管理员审核。

3. 实现完善的搜索机制，包括根据人名、更新活跃度、奖项、个人履历等进行筛选。

4. 通过邮箱或手机短信验证码支持新用户的注册和登录。

5. 实现网站的日间、夜间模式主题。

6. 支持一键部署方式，前端通过 Bun 或 Deno 任务脚本进行安装、构建与发布（如 `bun run build` 或 `deno task build`），后端可通过 `pyproject.toml` 或脚本管理。

7. 拥有详细的多个单元测试。

8. 具备标准化的注释（如 Python 的 Google 风格 Docstring）。

9. 配置规范的运行时多等级日志，并拥有一定的容灾预案，对数据进行自动化备份。

10. 限制媒体文件（图片等）的上传大小与次数，在数据库中存储路径而非文件本体。

11. 实现草稿箱以存储用户编辑的部分简介，并允许用户对草稿箱进行管理。

12. 接口文档明确，变量、文件、接口、路由命名与大小写规范统一。

### 可选

1. 通过算法匹配成员间的百科相似度，并在主页相应地推送对应成员以促进交流。

2. 实现更完善的安全防御机制，确保成员信息的隐私安全。

3. 在普通用户提交修改请求后，自动将请求推送到管理员邮箱以支持及时通知。

4. 接入小模型作为自然语言查询 AI，允许模糊查询与推荐相关用户信息。

5. 支持自定义纯色或图片页面主题。 

6. 实现 CI/CD。

7. 支持人脸识别匹配数据库中的人脸特征，通过上传图片跳转到对应用户页面。

8. 实现用户百科内的数据统计，形成词云或图表。

## 二、技术栈选择

1. 前端框架：推荐使用 React + Vite + TypeScript 实现，禁止新增 JavaScript 业务代码。

2. 前端运行时：前端不使用传统 Node.js 作为开发运行时，按项目要求在 Bun 或 Deno 中二选一。
	- 优先 Bun：对 Node 生态依赖兼容性要求高、迁移成本敏感时。
	- 选用 Deno：对权限隔离、运行时安全策略和标准库一致性有明确要求时。

3. 前端主题：推荐使用 TailwindCSS 或 React 相关框架实现。

4. 后端路由：使用 Nginx 限流并分发静态页面以避免高并发。

5. 后端服务器：推荐使用 FastAPI 作为基础框架，尽量避免 Django。根据并发量可选择性接入 Redis 作为缓存机制。

6. 数据库：推荐使用 ORM 框架对接 Sqlite 实现轻量部署。

7. 单元测试：推荐使用 Pytest 进行一键测试。

8. 类型检查：前端必须接入 TypeScript typechecker，作为 PR 合并与上线前的硬性检查项。

## 三、部署要求

尽可能减小内存占用与 CPU 功耗，实现轻量级部署。

部署依赖需要有脚本安装方式，前端推荐使用 Bun 或 Deno 作为运行时与任务执行器，后端使用 pip。

前端上线前必须先通过类型检查并完成构建，至少满足：
- `bun run typecheck && bun run build`
- 或 `deno task typecheck && deno task build`

对跨平台性不做要求，允许只在 Linux 上适配。

允许使用 Docker 和 Docker Compose 作为对接手段，严禁对外暴露不必要的 API。

## 四、版本管理要求

通过 `.gitignore` 文件限制版本上传内容。

1. 禁止上传构建产物（`build/`, `dist/`）与硬编码的第三方库（`node_modules/`）

2. 禁止上传本地缓存与配置产物（`__pycache__/`, `.idea/`, `.vscode/`）

3. 禁止上传密码和密钥（`.env`），如果涉及演示需求应上传含注释的 `.env.example` 作为示例。

4. 必须提交依赖锁文件（如 `bun.lockb` 或 `deno.lock`）以保证环境可复现。

## 五、开发路线

本项目建议分为五个阶段进行，前后端可在此框架下并行开发：

### 第一阶段：需求对齐与基础设施搭建（第 1-2 周）
- **产品设计**：使用 Figma 或墨刀绘制基础页面原型（首页、个人资料、搜索页）。
- **API 约定**：前后端成员共同制定并输出 RESTful API 接口文档（可使用 Apifox 或直接在 FastAPI 中写好 Schema 占位）。
- **数据库设计**：设计用户表、百科履历表、审核状态表，输出 ER 图。
- **工程初始化**：创建 Git 仓库，制定分支管理规范（如 GitFlow），初始化 React 前端与 Python 后端项目结构。

### 第二阶段：核心基础功能开发（第 3-4 周）
- **用户系统**：实现注册、登录（JWT 认证）、邮箱验证码发送机制。
- **个人百科 CRUD**：用户能够查看、创建、修改自己的基础履历信息，并保存为“草稿”或“待审核”状态。
- **前端展示**：完成首页的静态搭建、个人资料页的基础 UI 展示，适配日夜间模式；并建立 TypeScript 严格类型约束与基础 typecheck 流程。

### 第三阶段：权限管理与搜索功能（第 5-6 周）
- **审核工作流**：实现管理员鉴权，开发管理员后台或专门的审核页面，允许通过/驳回用户的履历更新请求。
- **高级搜索引擎**：后端实现多字段联合查询（姓名、履历关键字、奖项等），前端对接搜索框与结果展示列表。
- **文件上传**：实现证明材料（图片/PDF）的上传、存储与访问机制。

### 第四阶段：可选功能扩展与优化（第 7-8 周）
根据团队进度与能力，挑选可选要求进行开发（如：接入小模型进行模糊推荐、百科相似度匹配推送、自定义主题等）。
- **性能优化**：引入 Redis 缓存高频访问的首页数据或搜索热词（视并发测试结果而定）。
- **安全加固**：增加前端路由鉴权、后端接口的 XSS/SQL 注入防护、敏感信息（如手机号）的脱敏展示。

### 第五阶段：测试、部署与交付（第 9-10 周）
- **单元测试**：使用 Pytest 编写后端核心接口测试（尤其是鉴权与状态机流转），覆盖率达到 80% 以上。
- **部署上线**：编写 `Dockerfile` 与 `docker-compose.yml`（或自动化脚本），在 Linux 服务器上配置 Nginx 反向代理与 HTTPS 证书，完成生产环境部署；前端上线前必须完成 typecheck 与 build。
- **文档完善**：撰写《用户使用手册》与《项目维护指南》，进行结项复盘。
