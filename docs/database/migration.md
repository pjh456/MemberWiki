# 数据迁移与回滚规范

## 目标

保证数据库变更可追溯、可回滚、可重复执行。

## 迁移规则

1. 采用迁移工具（建议 Alembic）管理 schema 版本。
2. 每次迁移只做一类改动：建表、加字段、索引调整等，避免混合。
3. 迁移脚本命名建议：`YYYYMMDDHHMM_<change_summary>`。
4. 任何字段删除、类型变更都属于破坏性变更，必须提供 downgrade。

## 发布流程

1. 提交迁移脚本和对应文档更新（`schema.md`）。
2. 在测试环境执行升级与回滚双向验证。
3. 上线时先备份数据库，再执行迁移。
4. 迁移后执行关键查询与核心接口冒烟测试。

## 回滚策略

- 小版本回滚：使用最近一次 `downgrade` 回退。
- 大版本故障：恢复上线前数据库备份。
- 回滚后必须冻结新迁移，先完成故障复盘。

## 数据备份建议

- 每日全量备份一次，保留至少 7 天。
- 上线窗口前额外生成一次手动备份。
- 备份文件定期演练恢复，确保可用。
